Platform Overview

TeamSight is an engineering-metrics platform that collects, processes, and visualises KPI data across 165 employees organised into Teams and Scrums. It integrates with JIRA, GitHub, GitHub Copilot Metrics, and security-scanning tools to automatically evaluate up to 27 KPIs per employee per sprint period.

The platform follows a clean three-tier architecture: a Python KPI Engine that processes raw source data into structured CSV snapshots; a FastAPI REST backend that serves, schedules, and aggregates those snapshots; and a React/TypeScript single-page application that presents the data through drill-down dashboards (Team → Scrum → Employee).

All services are managed through dashboard/manage.sh. The backend (port 8000) also serves the pre-built React SPA as static files – no separate web-server is required.

KPI Engine Python · src/

The KPI Engine is a collection of Python scripts that transform raw issue-tracker and code-repository data into structured employee-level KPI measurements. Each KPI script is independently runnable and produces a standardised CSV output.

KppEvaluator.py acts as the orchestrator: it registers all KPI functions, resolves equivalent KPIs (where two KPIs share the same computation), coordinates input data loading, and dispatches execution to individual scripts.

Security Auth · RBAC · Transport

  • 🔐
    JWT Authentication (HS256) — All API endpoints except /auth/login and /health require a valid Bearer token. Tokens are signed with a configurable secret key and expire after 24 hours.
  • 🔒
    Password Hashing (bcrypt) — User passwords are never stored in plain text. passlib + bcrypt with auto-generated salts is used in AuthService.
  • 🛡️
    Role-Based Access Control (RBAC) — Four roles: Admin, API User, Viewer, Manager. Permission checks enforced via FastAPI dependencies (require_role, require_permission, require_admin).
  • 🌐
    CORS Middleware — Configurable allowed origins via cors_origins settings. Development allows all origins; production should restrict to the deployment domain.
  • 🔑
    Secrets via Environment — Sensitive values (JWT secret, API tokens) are loaded from dashboard/backend/.env via pydantic-settings, never hardcoded.
  • ✅
    Input Validation — All API request bodies and query parameters are validated by Pydantic models before reaching business logic, preventing injection-style attacks.

End-to-End Data Flow

  1. Data Fetch — jira_fetch.py, github_fetch.py, and copilot_metrics_fetch.py poll external APIs (on schedule or manually) and write raw snapshots to output/JIRAIssues.csv and output/github_commits.csv.
  2. KPI Computation — KppEvaluator dispatches the relevant kpp_k*.py script. The script joins raw data with Resources.csv and Roles.csv to compute Weekly/Monthly/Quarterly/Annual values per employee.
  3. CSV Persistence — The result DataFrame is appended (or replaced) to output/k*-data.csv with SAPID as the primary key per row.
  4. Cache Invalidation — watchdog detects the file change; the backend invalidates its in-memory DataFrame cache. Transition-history caches are re-warmed in background.
  5. API Request — The React SPA makes an authenticated GET request (with JWT) to e.g. /api/employee/kpis?name=John&period=Weekly.
  6. Backend Processing — FastAPI loads the relevant k*-data.csv via pandas, filters/aggregates by employee, applies ROG scoring thresholds from scoring_config.json, and serialises to JSON.
  7. Visualisation — The SPA receives the KPI payload and renders bar/radar/heatmap charts via ECharts and ROG status badges via Material UI.
