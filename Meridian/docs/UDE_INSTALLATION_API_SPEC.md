# UDE Installation Data API — Integration Specification

**Version:** 1.0  
**Date:** 22 April 2026  
**Owner:** TeamSight Platform Team  
**Audience:** External system integrators calling the UDE installation ingest endpoint

---

## 1. Overview

TeamSight exposes a REST API to receive UDE (Unified Development Environment) installation data from external systems. The API accepts a JSON payload describing one or more installation events and stores a full audit history of all installations per employee per device.

### Key design principles

- **One record = one installation event on one specific device.** The same employee reinstalling the same version on the same machine creates a new record with a new `install_event_id`.
- **An employee can have multiple devices.** Each device is tracked independently via `device_id`.
- **Full history is retained.** Records are never overwritten; they are appended/upserted by `install_event_id`. The calling system should set `status: superseded` on earlier records when a newer version is deployed.
- **Employee identity is via SAPID.** The `sapid` field must match the employee's SAPID as registered in the TeamSight resource directory.

## 2. Authentication

All API requests must include a **JWT Bearer token** in the `Authorization` header.

```
Authorization: Bearer <access_token>
```

### Obtaining a token

```
POST /api/auth/login
Content-Type: application/json
```

Request body:
```json
{
  "sapid": "api_user",
  "password": "<shared_api_password>"
}
```

Response:
```json
{
  "access_token": "eyJhbGci...",
  "refresh_token": "eyJhbGci...",
  "user": { ... }
}
```

- **Access tokens expire after 60 minutes.** Use the `refresh_token` to obtain a new one without re-authenticating.
- Contact the TeamSight administrator for the `api_user` credentials.

### Refreshing a token

```
POST /api/auth/refresh
Content-Type: application/json
```

Request body:
```json
{
  "refresh_token": "<refresh_token_from_login>"
}
```

Returns a new `access_token` and `refresh_token`.

## 4. UDE Installation Ingest Endpoint

### 4.1 Batch Ingest (JSON Body)

```
POST /api/ude/installations
Content-Type: application/json
Authorization: Bearer <access_token>
```

Accepts a batch of installation event records. Each record represents a single installation act on a single device for a single employee.

#### Request Schema

```json
{
  "report_date": "2026-04-22",
  "source_system": "UDE-Registry-v3",
  "installations": [
    {
      "install_event_id": "EVT-20260415-00123-001",
      "ude_id": "UDE-2024-00123",
      "sapid": "12345678",
      "employee_name": "Ananya A",
      "device_id": "LAPTOP-HCL-A1234",
      "device_label": "Primary Laptop",
      "version": "2.5.1",
      "release_name": "UDE-2026-Q1-GA",
      "release_date": "2026-03-01",
      "installed_date": "2026-04-15T10:30:00",
      "status": "installed"
    }
  ]
}
```

#### Top-Level Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `report_date` | string (`YYYY-MM-DD`) | Yes | The date this data snapshot was generated by the source system |
| `source_system` | string | No | Identifier of the calling system (e.g. `"UDE-Registry-v3"`) |
| `installations` | array | Yes | List of installation event objects (see below). A single call covering all employees (typically 300–600 records for a full organisation snapshot) is the recommended approach. The server accepts up to 5,000 records per request. |

#### Installation Object Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `install_event_id` | string | **Yes** | **Primary deduplication key.** Unique identifier for this specific installation event. Must be unique across all calls. If a record with this ID already exists, it is updated in place. |
| `ude_id` | string | **Yes** | The employee's UDE license / seat identifier as assigned by the UDE system. |
| `sapid` | string | **Yes** | Employee SAPID. Must match a registered employee in TeamSight. Records with an unrecognised SAPID are rejected with a warning (non-fatal). |
| `device_id` | string | **Yes** | Unique identifier of the physical machine (e.g. asset tag, hostname, MAC-derived ID). Must be stable across calls for the same machine. |
| `device_label` | string | No | Human-readable name for the device (e.g. `"Primary Laptop"`, `"Lab Workstation"`). Used for display only. |
| `version` | string | **Yes** | The UDE version string installed in this event (e.g. `"2.5.1"`). |
| `release_name` | string | No | Human-readable release label (e.g. `"UDE-2026-Q1-GA"`). |
| `release_date` | string (`YYYY-MM-DD`) | No | Date the release was published. |
| `installed_date` | string (ISO 8601) | **Yes** | Timestamp of when UDE was installed/updated on this device. Accepted formats: `YYYY-MM-DDTHH:MM:SS`, `YYYY-MM-DD HH:MM:SS`, `YYYY-MM-DD`. |
| `status` | string (enum) | No | Installation status. Default: `installed`. See §4.3 for allowed values. |
| `employee_name` | string | No | Employee display name from the source system. Advisory only — TeamSight resolves the canonical name from SAPID. |

#### 4.2 File Upload Variant

If the source system prefers to push a file rather than construct a JSON request body:

```
POST /api/ude/installations/upload
Content-Type: multipart/form-data
Authorization: Bearer <access_token>
```

Form field: `file` — a `.json` file containing the same schema as §4.1.

---

### 4.3 Status Values

| Value | Meaning |
|---|---|
| `installed` | Currently active installation on this device |
| `superseded` | This version was present but has since been replaced by a newer version on the same device |
| `uninstalled` | UDE has been explicitly removed from this device |
| `failed` | The installation attempt failed — UDE is not functional on this device |

**Recommended practice:** When pushing a new version for an employee on a specific device, include two records in the same batch:
1. The old `install_event_id` with `status: superseded`
2. A new `install_event_id` with `status: installed` and the new `version`

## 6. Response Schema

### Success — `200 OK`

All records accepted:

```json
{
  "status": "accepted",
  "received_at": "2026-04-22T10:01:05Z",
  "report_date": "2026-04-22",
  "total_records": 3,
  "accepted": 3,
  "skipped": 0,
  "errors": []
}
```

### Partial success — `207 Multi-Status`

Some records accepted, some rejected:

```json
{
  "status": "partial",
  "received_at": "2026-04-22T10:01:05Z",
  "report_date": "2026-04-22",
  "total_records": 5,
  "accepted": 3,
  "skipped": 2,
  "errors": [
    {
      "install_event_id": "EVT-20260422-00999-001",
      "sapid": "99999999",
      "reason": "SAPID not found in TeamSight resource directory"
    },
    {
      "install_event_id": "EVT-20260422-00888-001",
      "reason": "installed_date is missing or not a recognised date format"
    }
  ]
}
```

Skipped records do not affect accepted records — the rest of the batch is processed normally.

## 7. Error Responses

| HTTP Code | Scenario | Action |
|---|---|---|
| `400 Bad Request` | Malformed JSON, missing required top-level fields, `installations` is empty or exceeds 5,000 records | Fix the request payload |
| `401 Unauthorized` | Missing or expired Bearer token | Re-authenticate or refresh the token |
| `403 Forbidden` | Authenticated account lacks the required API role | Contact TeamSight admin to verify API user role |
| `422 Unprocessable Entity` | Field type mismatch (e.g. wrong data type, invalid enum value for `status`) | Check the field types against §4.1 |
| `500 Internal Server Error` | Unexpected server-side failure | Retry after a short delay; contact TeamSight admin if persistent |

## 10. Idempotency & Retry Behaviour

- Submitting the same `install_event_id` more than once is safe — the record is updated rather than duplicated.
- If a batch call fails mid-processing (`5xx`), the entire batch should be re-submitted. Records already accepted on the first attempt will be upserted without duplication.
- There is no rate limit currently enforced. **A single call covering all employees is the recommended pattern** — at ~300 employees with 1–2 devices each, the payload is approximately 200–600 records and ~200 KB, which is well within HTTP and server limits. Only split into multiple calls if sending complete multi-year history in one shot (>2,000 records).

## 11. Recommended Integration Pattern

```
1. On each sync cycle (e.g. nightly):
   a. Fetch the current installation snapshot from the UDE registry.
   b. For any version upgrades since the last push:
      - Include the old install_event_id with status: superseded
      - Include the new install_event_id with status: installed
   c. POST the full delta (or full snapshot) to /api/ude/installations.
   d. Check the response for any skipped records and investigate rejected SAPIDs.

2. Token management:
   - Obtain a token at process start.
   - If a 401 is received, call /api/auth/refresh and retry once.
   - If refresh also fails, re-authenticate via /api/auth/login.
```
